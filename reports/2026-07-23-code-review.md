## 1. UX & Flow Quality (priority)

### Scan → Label → Confirm → Results → Wine Intel

- **[Medium] OCR failures are swallowed silently; Confirm can reopen with stale wine data.** `app/label/camera.tsx:92-95` and `app/(tabs)/scan.tsx:128-133` call `setError(...)` on OCR failure but still navigate to `/label/confirm` regardless. `app/label/confirm.tsx` never reads `labelStore.error`, so no banner/toast appears, and `useLabelStore.setError` doesn't clear `wineDetails`. Scenario: scan wine A successfully and confirm it, then scan wine B and the OCR call fails (network blip) — Confirm silently reopens still pre-filled with wine A's producer/region/vintage, with copy that says "Check the details we extracted" as if fresh extraction happened. Fix: surface `labelStore.error` as a dismissible banner in `confirm.tsx`, and clear stale `wineDetails` on a failed scan so the form opens blank.
- **[Low] "Scan Again" mislabels a modality switch for the Upload-photo entry path.** `app/label/confirm.tsx:351-355` — for the "Upload A Wine Label" path (gallery picker, never touched the camera), the retry button still reads "Scan Again" and routes to `/label/camera` (live camera), not back to the photo picker the user actually used.
- **[Low] Two full modal layers stand between "Wine Label" and the camera.** `app/(tabs)/scan.tsx:177,194-233` — tapping "Wine Label" opens a sign-in soft-gate (if guest), then a "Generate Wine Intel" chooser with 4 buttons (Scan Label / Upload / Manual / Cancel) before the camera opens. For what's almost certainly the dominant action (just scan a label), that's 2-3 taps of pure chrome. Consider making "Scan Label" the default one-tap affordance with Upload/Manual as a secondary link.

### Add-a-wine surfaces (rack vs. storage location vs. bin cell)

- **[Medium] Three different UI patterns for functionally the same action, with no shared vocabulary.** Rack (`app/cellar/rack/[rackId].tsx:1567-1588`) offers a 4-option chooser sheet including "Select from Cellar List" and confirms placement with an inline "Saved to rack & Cellar List" banner. Storage location (`app/cellar/storage-location/[id].tsx:493-508`) offers only 3 buttons — no "Select from Cellar List" — and gives **no confirmation at all** after a successful add (`app/label/results.tsx:502-546`); the only sign it worked is the wine silently appearing in the list. Bin cell (`app/cellar/bin/cell/[cellId].tsx:244-283`) is a completely separate inline form that never leaves the screen, has its own "Scan label to fill" button, and also closes with zero confirmation. A wine already in the cellar/wishlist can be picked straight into a rack slot but not into a storage location or bin cell — the user must re-scan/re-type it, risking a near-duplicate row. Recommend one shared "add to X" chooser component (same 3-4 options including "Select from Cellar List") and the same post-save confirmation treatment across all three.
- **[Low] Bin cell's "over capacity" warning and rack's "bottle size mismatch" warning use different copy for the same soft-warn-and-proceed pattern.** `app/cellar/bin/cell/[cellId].tsx:105-120` ("Over capacity" / "Save anyway") vs. `app/cellar/rack/[rackId].tsx:704-712` / `app/label/results.tsx:556-566` ("Bottle size mismatch" / "Place anyway"). Same product pattern, inconsistent voice.
- **[Low] Restaurant review empty-state copy references a button that doesn't exist.** `app/restaurants/reviews.tsx:590` tells the user to tap "Review this restaurant" — no button with that text exists anywhere in the app (the actual buttons read "+ Review/Edit Restaurant" in `app/scan/results.tsx:600` and "Add +" in `reviews.tsx:599`). Update the empty-state copy to match the live button text.

### Cellar screens (Full List / Racks overview / Wine Reviews)

- **[High] "Your Wine Reviews," "Your Cookbook," and "Your Recipe Reviews" render a blank screen while loading — no spinner.** `app/wines/chosen.tsx:719`, `app/chef/archive.tsx:298`, `app/recipes/chosen.tsx:148` all use `isLoading ? null : ...`. On a slow connection this reads as a frozen/broken screen. Fix: show a centered `ActivityIndicator`, as `app/cellar/list.tsx` and `app/cellar/lineups.tsx` already do correctly.
- **[High] The "Wish List Wines" filter is dead — no UI ever opens it.** `app/wines/chosen.tsx:361-370` defines a "Collection" dropdown (`PORTFOLIO_OPTIONS`: All / Cellar / Restaurant / Wish List) fully wired into the filter logic at `:306-321` and `dropdownConfig('type')` at `:380`. But the rendered filter-chip row (`:760-780`) only wires up Sort, Favourites, and City — confirmed by reading the JSX directly, there is no chip anywhere that calls `setOpenDropdown('type')`. A user can never actually select "Wish List Wines" to filter down to it, even though the underlying filtering fully supports it. This is the single most impactful UX gap found for the Wishlist surface. Fix: add the missing "Collection" chip alongside Sort/Favourites/City.
- **[High] Full Cellar List's error state tells the user to "pull down to refresh" — no pull-to-refresh exists.** `app/cellar/list.tsx:873`: *"Check your connection and pull down to refresh. Your wines are safe."* There is no `RefreshControl` anywhere in the file. A user who hits this error has no way to retry short of leaving and re-entering the screen. Fix: either add `RefreshControl`, or swap the copy for a real "Try again" button, as `app/cellar/lineups.tsx:344-346` already does correctly.
- **[Medium] Racks/Bins/Other-Storage overview can't distinguish "no storage yet" from "fetch failed."** `useRacks()` (`src/hooks/useRacks.ts:9-13`) never surfaces `isError`, so `app/cellar/racks.tsx` has no error branch — a failed fetch renders identically to a brand-new user with nothing set up. `app/cellar/list.tsx:866-887` deliberately guards against this exact class of bug; `racks.tsx` doesn't.
- **[Medium] Inconsistent "+Add" placement between sibling Cellar screens.** `app/cellar/list.tsx:702-707` and `app/cellar/lineups.tsx:329-331` place "+ Add" as a persistent header link. `app/cellar/racks.tsx:224-226,252-254` instead buries it at the *end* of each horizontal carousel, requiring a full swipe-through as the list grows. Same action, two patterns, on screens one tap apart.
- **[Low] Long-press-only affordances have no visual hint they exist.** `app/(tabs)/cellar.tsx:89-92` (long-press "Archive a Night" for an explainer) and `app/(tabs)/chef.tsx:109,120` (long-press either action button to revisit the last result) hide real functionality behind an undiscoverable gesture, with no "hold for details" label or icon, unlike the `HelpButton` pattern used elsewhere on the same screens.

### Home hub / global navigation

- **[Medium] No single-tap way back to the branded Home hub once you've left it.** `app/home.tsx` is the real post-login landing screen (hosts personality nudges, the hamburger shortcut menu, the 2×2 destination grid) but `AppBottomBar.TABS` (`src/components/AppBottomBar.tsx:10-16`) has no "Home" entry, and the bottom bar is explicitly hidden on `/home` itself (`HIDDEN_FIRST`, `:21-23`). The header logo (`VinsterHeader.tsx:12`) links to `/about`, not `/home`. Once a user has branched into any tab, only OS back-gesture/hardware-back can return to Home — not discoverable, not reliable once navigation has branched. Consider adding a Home entry to the bottom bar or making the logo mark a tap/long-press-to-home shortcut.

### Onboarding

- **[Low] "Back through carousel" is internal jargon exposed to the user.** `app/onboarding.tsx:245`. Suggest "Watch the intro again" or similar.

### What's working well

- The core Scan → Confirm → Results replace-discipline is sound: every step uses `router.replace` rather than `push`, keeping the native stack at constant depth, so hardware back/swipe-back correctly returns to the true origin screen in every flow traced.
- `backTo` threads correctly end-to-end through `label/camera → label/confirm → label/results` for every current call site, including retries.
- Community "coming soon" screens (`community/profile.tsx`, `community/[category].tsx`) are honestly disabled (`pointerEvents="none"` + reduced opacity) rather than half-wired — a good pattern other preview surfaces could be checked against.
- Auth screens, age-gate, onboarding tour, and `chef/find-pairing.tsx` / `review-requirements.tsx` / `results.tsx` all have real loading indicators, clear error copy, and consistent modal patterns.

## 2. Navigation & Back-Command Issues

- **[High] Stale `pendingSlot` can silently hijack an unrelated wine save into an abandoned rack slot.** `app/label/results.tsx:547` gates its rack-placement branch on `if (pendingSlot)` alone. Compare `app/label/confirm.tsx:124`, which gates the equivalent branch on `pendingSlot && context === 'place'` — with an inline comment explicitly calling out "stale pendingSlot hijacking an unrelated add" as the bug being guarded against. `results.tsx` never got the same guard, and `pendingSlot` is only cleared on a successful placement or a full account-switch wipe (`src/stores/clearAllStores.ts` explicitly preserves it across a normal flow reset). **Verified directly**: `results.tsx:547` reads `if (pendingSlot) {` with no `context` check, while the storage-location branch immediately above it (`:502`) correctly requires `pendingStorageLocationId && context === 'add-location'`. Failure scenario: tap an empty rack slot → "Manual Input" → cancel out via Confirm's Cancel button (which also never clears `pendingSlot`) → go to Cellar List → "Add Wine" → scan/enter an unrelated wine and save. The new wine is silently placed into the abandoned rack slot and the user is redirected to that rack instead of the Cellar List, with no explanation. Fix: add `&& context === 'place'` to the condition at `results.tsx:547`, mirroring `confirm.tsx:124`.
- **[High] "Discard" on the Wine Intel card disagrees with the header back arrow directly above it.** `app/label/results.tsx:1096` still uses the pre-fix default `router.replace(backTo ? decodeURIComponent(backTo) as any : '/(tabs)/cellar')`, while the header back arrow (`:958-968`, fixed in commit `575c9f7`) correctly defaults an intel-only flow back to `/(tabs)/scan`. Scenario: from Scan tab, long-press "Wine Label" → "View Last Result" (`app/(tabs)/scan.tsx:112`, no `backTo` passed) → on the resulting Wine Intel card, the gold "←" arrow returns to Scan correctly, but the "Discard" button — whose whole purpose is "leave this screen" — instead sends the user to the Cellar tab. Two exits, same screen, same missing-`backTo` condition, two different destinations.
- **[Medium] "Couldn't generate intel" prompt's "Check details" button ignores `backTo`/context entirely.** `app/label/results.tsx:986`: `onEdit={() => router.replace('/(tabs)/cellar')}` — hardcoded, doesn't consult `backTo` or `isIntelOnlyFlow` at all, unlike every other exit on the screen. Reachable whenever a Scan-tab-originated intel result comes back with no critic score and no estimated value (a garbled OCR read). Same wrong-destination symptom as the Discard finding above.
- **[Medium] Restaurant review: reopening a deep-linked review and then tapping "Edit Wine"/"Review Wine" can silently do nothing.** `app/restaurants/reviews.tsx:269-281` (`closeRestaurantReview`) calls `router.back()` whenever the modal was reopened via the `?openSession=` deep link. But `onReviewWine`/`onEditWine` (`:754-776`) call `closeRestaurantReview()` and then, in the same synchronous handler, set `editWineIdentity`/`editingWine` state **on the screen instance that `router.back()` just told the navigator to pop**. Scenario: Your Restaurants → open a visit → "View Wine Intel" on a bottle → back (correctly routes via `backTo` to `reviews.tsx?openSession=...`, reopening the same visit) → tap "Edit Wine" or "Add/View Review" on a bottle. The screen pops back to a *previous*, buried `reviews.tsx` instance with its own separate `editingWine` state (`null`) — the requested edit modal never appears; the tap looks like it did nothing. Fix: don't call `router.back()` when the close is immediately followed by opening a sibling in-page modal.
- **[Medium] "View Wine Intel" failure from inside a restaurant review dead-ends the user outside the review.** `app/restaurants/reviews.tsx:777-789` closes the review modal (`editing(null)`) *before* the async `generateWineIntel` call starts; on failure it only shows a generic alert and never reopens the review the user was in. Edits are safe (autosave on unmount, `RestaurantReviewModal.tsx:146-165`), but the user is dropped on the bare restaurant list and has to re-find and reopen the visit. Fix: reopen the same `editing` item on failure instead of leaving it closed.
- **[Medium] Fragile fixed-delay `setTimeout` used to sequence intel navigation past a closing modal.** `app/restaurants/reviews.tsx:780-787` — a 320ms delay is a guess at the review modal's slide-close animation duration; the inline comment documents this as a workaround for navigation being "swallowed" while the modal is mid-dismiss. On a slower device/dropped frame the delay may not be enough, reintroducing the exact bug it was added to fix. Prefer an event-driven trigger (`onDismiss`/modal-hide callback) over a magic-number delay.
- **[Medium] Two independent full-screen modals can appear stacked on the Wine Intel card for the exact same scenario.** `app/label/results.tsx:983-988` (`NoIntelPrompt`, shown when intel came back empty) and the separate `promptAddToLabelLibrary()` effect (`:251-279`, guarded only by `isIntelOnlyFlow && fresh === '1'`, not by whether intel actually populated) can both become visible on first paint for a garbled-OCR scan, since `confirm.tsx:170` always sets `fresh=1` for any intel-context confirm regardless of outcome. Neither modal is aware of the other.
- **[Medium] Storage-location "add a wine" leaves a duplicate screen on the stack — Back needs two taps to actually leave.** The chain `storage-location/[id].tsx:130-163` (`router.push`) → `label/camera.tsx:91` (`replace`) → `label/confirm.tsx:153` (`replace`) → `label/results.tsx:544` (`replace` back to `/cellar/storage-location/{id}`) lands a second, brand-new instance of the same screen on top of the original un-popped one. The header back button is a plain `router.back()` (`storage-location/[id].tsx:479`), so the first tap only pops to the buried duplicate (visually identical — same location, same content), requiring a second tap to actually leave. The equivalent rack flow hits the same underlying shape but papers over it with a deliberate `dismissTo('/cellar/racks')` (`app/cellar/rack/[rackId].tsx:412-421`, documented inline) that collapses both duplicates in one tap — storage-location never got that treatment. Fix: apply the same `dismissTo`-based collapse to storage-location's back button, or make `results.tsx:544`'s final replace a `dismissTo` back to the hub.
- **[Medium] Label Scan Library's own Upload button was not included in the "upload bounce" fix.** `app/scan/archive.tsx:129-134` (`handleUpload`, wired at `:486`) still does `setAddOpen(false)` immediately followed by `ImagePicker.launchImageLibraryAsync(...)`, with no deferral. Commit `575c9f7` fixed exactly this race (native picker launched the same tick a custom `<Modal>` starts closing, which can silently swallow the picker call) in `app/(tabs)/scan.tsx` via a `setTimeout(handleUploadLabel, 350)` wrapper, but `archive.tsx` was not touched by that commit and still has the bug. Scenario: Label Scan Library → "+Add" → "Upload a Photo" → the picker can silently never present, leaving the user stuck with no feedback.

### What's working well

- The `pendingStorageLocationId` branch in `results.tsx:502` is correctly gated on `context === 'add-location'` — the same class of bug fixed there was simply never mirrored onto the `pendingSlot` branch two lines below it (see the High finding above).
- The one "upload bounce" fix that *was* applied (`app/(tabs)/scan.tsx:212`) is implemented cleanly with a clear inline rationale — no leftover dead code, no double-deferral.
- Wine Bins hit-testing, capacity thresholds, and scan-to-fill scoping are all correctly self-contained per-cell/per-bin with no cross-navigation leakage (see Section 4).

## 3. Other Home Storage (dedicated deep-dive)

Files: `app/cellar/storage-location/[id].tsx`, `src/api/storageLocations.ts`, `app/cellar/storage-location/new.tsx`, plus every call site of `caseKindLabel`/`deleteEmptyCasesForLocation` and the cross-cutting paths in `app/label/results.tsx` and `app/cellar/[wineId].tsx` that feed wines into this screen.

Seven findings from the 2026-07-21 report (S1/S2, D1-D4, U1) were re-verified this run and are now **fixed** on `main`: delete-location cache invalidation, wishlist exclusion from counts, empty-case rendering, case-create rollback, `case_id` clearing on remove, location-create rollback, and location rename. They are not repeated below.

### Data-integrity issues

- **[High] Archiving or hard-deleting a wine from its own detail screen skips all case cleanup.** `app/cellar/[wineId].tsx:578-605` (drink-to-zero archive) and `:784-819` (hard delete) never clear `case_id` and never call `deleteEmptyCasesForLocation`, unlike every equivalent action reachable from the storage-location screen itself, which always calls `assignWineToCase(id, null)` first (`storage-location/[id].tsx:249,305,314`). Scenario: file a wine into a case, open its wine-detail card (not the location's own long-press menu), archive or hard-delete it. The now-empty case isn't swept until the location screen happens to be freshly re-mounted (its own `useEffect` sweep, `[id].tsx:95-100`); until then, within react-query's 60s `staleTime`, the location can show a stale "ghost" wine and/or an undeletable zero-wine case.
- **[High] A single-wine case (`owc`/`non_owc`) can be corrupted with an unrelated second wine.** `storage-location/[id].tsx:356-374` and `app/label/results.tsx:515-535` — "Add another to this case" always offers Scan/Upload/Manual regardless of `case.kind`, and nothing checks that the newly scanned wine matches the wine already in an `owc`/`non_owc` case before `assignWineToCase` runs. Scenario: open an "OWC Complete" case's menu → Scan a Label → scan a totally different bottle → it's saved as a new `cellar_wines` row sharing that case's `case_id`, even though the kind means "one wine, complete case." `confirmAddBottles` (`[id].tsx:376-393`) then does `wines.find(w => w.case_id === c.id)` — with two wines sharing a case_id it silently picks whichever sorts first, so "Add more bottles of this wine" can bump the wrong wine's quantity with no indication to the user.
- **[Medium] Bin placement isn't checked before offering "Add to Location" — a wine can be filed into a bin cell and a storage location simultaneously.** `app/cellar/[wineId].tsx:1444` only hides "Add to Location" when there's no rack/other-location membership; there is no `bin_cell_id`/`wineBins` check anywhere in the file (confirmed via a zero-hit grep for `bin_cell_id` in this file). `fileInStorageLocation` (`:451-461`) sets `storage_location_id` without clearing `bin_cell_id`. A wine placed in a bin cell can then also be filed into a storage location, double-counting inventory in both the bin's cell count and the location's bottle list — the same class of bug the code already guards against for racks (`results.tsx:504-507`, comment "S3") but never extended to bins.

### Cache-invalidation issues

- **[Medium] "Add more bottles of this wine" never invalidates the `storage-locations` list query.** `storage-location/[id].tsx:385-392` invalidates `['storage-location-wines', id]` and `['cellar']` only. The Home Storage carousel's per-card bottle count (`app/cellar/racks.tsx:104,249`, sourced from `['storage-locations', userId]`) is left stale for up to the global 60s `staleTime`.
- **[Medium] Same gap restated for cache purposes**: `[wineId].tsx:578-605` and `:784-819` (the archive/delete paths from Finding 1 above) are also missing invalidation of `['storage-locations']`, `['storage-location-wines', locId]`, and `['storage-location-cases', locId]` — the `['cellar-locations']` key invalidated elsewhere in that file (lines 442, 641, 668, 810) is an unrelated custom-filter feature, not this table.

### UX issues

- **[Medium] "Cases List" view mode ignores the Maturity filter and Search entirely.** `storage-location/[id].tsx:603-621` computes each case's displayed bottle count from the raw `wines` array, not `filtered` (the maturity/search/cases-filtered array used everywhere else at `:110-124`), and `caseGroups` (`:432-434`) is filtered only by `casesFilter`. Set Maturity = "Peak" and switch to Cases List — every case still shows with its full unfiltered count, as if no filter were active. This is the exact composition failure the review brief asked to check for.
- **[Low] Cases filter has no option for legacy `kind='single'` cases.** `CASE_FILTER_OPTIONS` (`storage-location/[id].tsx:40-45`) only lists `owc`/`non_owc`/`mixed`. Migration 073 deliberately keeps `single` valid for pre-073 rows with no backfill; `caseKindLabel` renders them fine (falls through to "Case"), but a user can never isolate them via the Cases filter — only "All cases" ever surfaces them.
- **[Low] Archive/Delete from the per-wine popup always acts on full quantity — no partial-quantity option, unlike the rack screen.** `storage-location/[id].tsx:309-324` calls `archiveCellarWine`/`deleteCellarWine` (`src/api/cellar.ts:107-120,201-212`), neither of which takes a quantity. The rack screen (`app/cellar/rack/[rackId].tsx:116-120,811-826,934-940`) asks "how many of your N bottles?" before archiving/deleting a multi-bottle wine. Tapping Archive on a 12-bottle OWC case archives all 12 at once with no partial-pull option. (This matches the Full Cellar List's own bulk-archive behavior, so it's a pre-existing app-wide gap rather than something newly introduced — but it is the one inconsistency with the rack/fridge pattern the review brief specifically asked about.)

### Other

- **[Low, maintenance risk] The entire multi-select bulk-action system on this screen is dead code.** `enterSelect` (`storage-location/[id].tsx:199`) has zero callers anywhere in the file (confirmed by grep) — the comment at `:294-295` explains the per-wine long-press popup replaced it, but `selectMode`/`selectedIds`/`toggleSelected`/`exitSelect`/`runBulk`/`confirmArchiveSelected`/`confirmDeleteSelected`/`confirmRemoveSelected` (`:74-278`), the checkbox rendering (`:448-450`), and the floating select bar (`:658-678`) are all still fully wired but unreachable. A future reader will reasonably assume there's a working second interaction path — there isn't. Recommend deleting the dead code rather than leaving it live-looking.
- **[Low] Stale, self-contradicting comment.** `storage-location/[id].tsx:181-183` — the comment above the delete-location handler still describes a cache-invalidation bug ("this one didn't [invalidate], leaving the deleted card... (S1)") that the code immediately below it (`:184-188`) already fixes. Leftover from the 07-21 fix; will mislead the next person grepping this file for cache bugs.

### What's working well

- FK design is sound: `storage_cases.storage_location_id ... on delete cascade` and `cellar_wines.storage_location_id / case_id ... on delete set null` mean deleting a location cannot leave a DB-level orphan by itself.
- Every action reachable from `openWineActions`/`runBulk`/`runSingle` correctly scopes to the long-pressed/selected wine id via plain closures — no stale-selection bug found.
- `caseKindLabel` is used consistently everywhere a case kind is rendered — no raw enum leaks found anywhere in the repo.
- Wishlist exclusion is correctly applied to both the wine list and the bottle count (`storageLocations.ts:16,24,96`), matching the rack path.
- The create-location photo flow is clean: no `allowsEditing`, `resizeMode="contain"` at both preview and detail display, no dead cropping code.
- Destructive confirmations (delete location, dissolve case, archive/delete/remove wine) all have clear, accurate copy about what stays vs. what leaves.
- The mount-time `deleteEmptyCasesForLocation` sweep is a real server round-trip per visit (not gated on client cache staleness), so it reliably self-heals orphaned-empty cases the next time a location is actually reopened — its only gap is locations the user hasn't reopened yet (see the two High findings above).

## 4. Today's Work (per-area findings)

### Wine Bins — `supabase/migrations/072_wine_bins.sql`, `src/api/bins.ts`, `app/cellar/bin/[binId].tsx`, `app/cellar/bin/cell/[cellId].tsx`

- **Math verified correct.** `binDiamondCount(a,d) = a*d + max(0,a-1)*max(0,d-1)` and `binTriangleCount(a,d) = 2*(a+d)` were hand-traced against the actual `tessellate()` rendering logic (`app/cellar/bin/[binId].tsx:20-35`) for across=3/down=2, 1/1, and 1/2. All three cases matched exactly (e.g. 3×2 → 8 full diamonds + 10 clipped triangles, matching `3*2+2*1=8` and `2*(3+2)=10`). This was previously buggy (fixed in commit `b06cb33`) and the current formulas are correct.
- Hit-testing (`[binId].tsx:90-102`, nearest-centre search) is mathematically sound for a square lattice — every point inside a visible (frame-clipped) triangle is still closer to its own off-screen centre than to any neighbour's, verified by example. No desync between the `Pressable`'s coordinate frame and the children's layout frame.
- Over-capacity warning (`cell/[cellId].tsx:105-120`) reads `cell.capacity` straight from the DB row (already diamond/triangle-correct via `buildBinCells`) with a correct strict `>` threshold — no recomputation that could drift.
- Scan-to-fill (`cell/[cellId].tsx:73-97,99-183`) is correctly scoped to the route's `cellId` for both duplicate-matching and the new-row insert — no cross-cell/cross-bin leakage.
- **[Low] `app/cellar/bin/new.tsx` is dead, unmaintained code.** Nothing links to `/cellar/bin/new` (only `/cellar/bin/resize` is referenced, from `app/cellar/racks.tsx:126`). It was the original bin-creation flow, superseded by the corner-pull sizer in `resize.tsx` (commit `c776a73`) but never deleted. Its stepper max (12) has already drifted from `resize.tsx`'s `MAX=10`, evidence it's not being kept in sync. Recommend deleting it.
- **[Low] Delete-gesture inconsistency: long-press-to-delete works on rack/fridge cards but not bin cards.** `app/cellar/racks.tsx:25-40,67-84` wires `onLongPress` for racks; the bin carousel card (`:212-223`) is a plain `TouchableOpacity` with no `onLongPress`. (Bins are at least consistent with Other Home Storage locations, which also require opening the detail screen to delete — so this is an inconsistency with racks specifically, not with everything.)

### Cellar Stats — `app/cellar/stats.tsx`, `src/components/WineValueEditorModal.tsx`

- **[High] "Change since purchase" can silently become a subset figure with zero UI indication, in a case the code doesn't even detect.** `changePartial` (`stats.tsx:119`, **verified**: `const changePartial = changeEntries.length > 0 && (winesNoPurchase.length > 0 || winesNeedingEstimate.length > 0);`) only fires the "this % excludes some wines" caption when wines are *missing* a value. It does not detect wines that have both values but were excluded for a currency mismatch (`pc !== vc`, line 106 — this exclusion itself is correct behavior, not a bug). If every wine in the cellar has both fields populated but some pairs are in different currencies, `changePartial` is `false`, the caption never renders, and the user sees a bare "▲ 7.2%" with nothing indicating it's computed on a subset. Root cause traced to `WineValueEditorModal.tsx:52-64`: unlike `app/cellar/[wineId].tsx:694-697,866-867` (which carefully preserve/seed a wine's existing counterpart-field currency), the modal always stamps whatever the caller's *current default currency* is onto whichever field it's editing, with no awareness of the wine's other field — this is what produces the mismatches in the first place. Fix: broaden `changePartial` to also fire when `matchedCount` is less than the count of wines with both fields populated (i.e., detect currency-mismatch exclusions separately), and make `WineValueEditorModal` write a currency that matches the wine's existing counterpart field when one exists.
- **[Medium] Tapping "Close" mid-save does not cancel the in-flight save — it just hides the modal.** `WineValueEditorModal.tsx:107-110` disables Save while `saving` but the Close button has no such guard; `handleSave` (`:46-73`) writes wines sequentially in an `await`-ed loop. If a user taps Close after 2 of 5 writes have completed (believing this cancels), the remaining writes continue in the background and `onSaved()` still fires, invalidating caches silently after the modal is already hidden. A user can end up with values saved that they thought they'd cancelled.
- **[Medium] A partial write failure never invalidates the cache, leaving already-saved values stale in the UI.** Same `handleSave` loop — if write #3 of 5 throws, the `catch` only shows an alert; `onSaved()` (the only place that invalidates `['cellar', ...]`) is never called, so wines #1-2 are persisted to Supabase but Stats keeps showing pre-edit totals for them until an unrelated full save or a screen remount triggers a fresh fetch.
- **[Low] Zero-cost wines are counted in the "compares N wines" caption but silently excluded from the % itself.** `changeEntries` (`stats.tsx:113-114`) filters out any currency group whose summed purchase total is `<= 0`, but `matchedCount` (used in the caption text) still counts those wines — the count and the rendered percentages can describe different sets.

### Wine Review Dedup (historically delicate) — `src/components/AddChosenWineModal.tsx`, `src/components/ChosenWineModal.tsx`, `app/wines/chosen.tsx`

- **[Medium, confirmed] Same-day vs. earlier-date classification compares UTC calendar dates, not local ones.** **Verified directly**: `AddChosenWineModal.tsx:87-89` and the equivalent in `ChosenWineModal.tsx` both compute `new Date(existing.chosen_at).toISOString().slice(0,10)` and `new Date().toISOString().slice(0,10)` — `toISOString()` always renders the UTC date. Not data loss, but defeats the "awaiting" signal for a legitimately pending item. Worth a product decision on whether this is acceptable.

## 6. Supabase / RLS / Data Integrity

- **RLS spot-checked directly on the two newest tables and found correct.** `bin_cells` (migration 072): `enable row level security` + a policy scoping to rows whose parent `wine_racks.user_id = auth.uid()`, using `for all` with only a `using` clause — per Postgres semantics this correctly applies the same check to inserts/updates when no separate `with check` is given, so this is not a gap. `storage_cases` (migration 069, still governing the case-kind expansion in 073): `enable row level security` + explicit `using (auth.uid() = user_id) with check (auth.uid() = user_id)`. Both are sound. RLS on other tables (`cellar_wines`, `wine_racks`, `storage_locations`, etc.) was not re-audited this run — no evidence of a regression was found in any of the code paths reviewed, but a full RLS audit across all tables was outside this review's file scope and should be tracked separately if desired.
- **Foreign-key cascade design is sound** for the areas reviewed: `bin_cells.bin_id ... on delete cascade`, `cellar_wines.bin_cell_id ... on delete set null`, `storage_cases.storage_location_id ... on delete cascade`, `cellar_wines.storage_location_id / case_id ... on delete set null`. No route exists that could leave a `cellar_wines` row pointing at a deleted bin, cell, case, or location purely at the DB level.
- **Application-level orphaning still possible despite sound FKs** — this is a code-logic gap, not a schema gap. Two paths (detailed in Section 3) can leave a **non-deleted but effectively orphaned** state that the DB's cascades don't cover because nothing deletes the parent: archiving/deleting a wine from its detail screen leaves a zero-member case sitting live (not cleaned until the location screen is next mounted), and cross-case-kind wine additions can put two unrelated wines under one `owc`/`non_owc` case_id with no schema-level constraint to prevent it (the `kind` check constraint only validates the enum value, not cardinality).
- **Migrations 072 and 073 are confirmed already applied to the live database** per standing instructions — none of their columns/tables were flagged as missing; all findings above concern application code that reads/writes them, not the migrations themselves.
- **No RLS or data-integrity issues were found in the wine-review dedup tables** (`chosen_wines`) — all deletion/update paths in the three dedup files are single-table, id-scoped, and non-cascading (Section 4).
